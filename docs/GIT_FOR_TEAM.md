# Работа с репозиторием для команды

Краткое руководство для коллег: как получить код, не сломать историю и согласовать выгрузку из 1С с Git.

## 1. Доступ

1. Убедитесь, что у вас есть доступ к репозиторию на GitHub (приглашение от владельца, SSH-ключ или Personal Access Token для HTTPS).
2. Клонируйте репозиторий:

```text
git clone <URL-репозитория>
cd DOC_FSK_DEV
```

Рекомендуется **SSH** (`git@github.com:...`), если настроен ключ.

## 2. Что в репозитории важно для расширения

Основной исходный код расширения в каталоге:

`DOCFSKDEVII.ИИАнализДокументовOllama/`

- `src/` — выгрузка EDT (XML метаданных и BSL).
- Готовый файл **`releases/ИИАнализДокументовOllama.cfe`** — актуальная сборка для загрузки в конфигуратор; при желании дублируйте его во **вкладке Releases** на GitHub.

Документация в корне и в `docs/`:

- `README.md` — обзор и ссылки.
- `docs/INSTRUCTIONS.md` — установка и использование.
- `docs/OLLAMA_SERVER.md` — развёртывание Ollama на сервере/ВМ.
- `CHANGELOG.md` — история версий.

## 3. Типовой цикл: правки в Конфигураторе / EDT и коммит

1. Сделайте ветку от актуальной **`main`** (или от согласованной релизной ветки):

```text
git checkout main
git pull
git checkout -b feature/краткое-описание
```

2. Внесите изменения в расширение в 1С.
3. **Выгрузите** изменения в файлы проекта (EDT: синхронизация с Git; Конфигуратор: выгрузка в файлы в каталог расширения, если так принято у команды).
4. Просмотрите diff:

```text
git status
git diff
```

5. Добавляйте **только** нужные пути (избегайте случайного `git add .` всего диска с длинными путями основной конфигурации, если она в том же рабочем каталоге).

```text
git add DOCFSKDEVII.ИИАнализДокументовOllama/
git add docs/ README.md CHANGELOG.md
```

6. Коммит с понятным сообщением на русском или английском, одна мысль — один коммит по возможности.

```text
git commit -m "ИАДО: кратко что сделано"
git push -u origin feature/краткое-описание
```

7. Создайте **Pull Request** в `main` (или попросите ответственного смержить).

## 4. Соглашения

- Не коммитить `.cursor/`, логи сборки, локальные `*.cfe`, если это не договорённый релизный артефакт в отдельной ветке.
- Крупные бинарники и полная типовая конфигурация **не обязаны** лежать в Git — уточняйте у владельца репозитория.
- Версию расширения (`Configuration.mdo`, поле `version`) поднимайте осознанно и отражайте в `CHANGELOG.md`.

## 5. Релизы для коллег без Git

Для пользователей, которым нужен только результат:

1. Откройте страницу **Releases** репозитория (ссылка в `README.md`).
2. Скачайте последний `.cfe`.
3. Подключите в конфигураторе как расширение и обновите БД.

Подробности установки — в `docs/INSTRUCTIONS.md`.

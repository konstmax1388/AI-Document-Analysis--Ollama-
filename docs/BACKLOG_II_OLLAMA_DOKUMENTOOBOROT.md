# Бэклог: ИИ-анализ файлов документов через Ollama

**Расширение:** `DOCFSKDEVII.ИИАнализДокументовOllama`  
**Платформа:** 1С:Документооборот 3.x (ориентир 3.0.19+), 1С:Предприятие 8.3.23+  
**Документ плана:** 2026-04-01 (актуализация дорожной карты и приоритетов)

---

## 1. Цель продукта

Сделать **устойчивый и управляемый** процесс ИИ-анализа файлов документов в ДО:

- запуск из карточки документа (основной сценарий, не ломать);
- выбор эксперта по бизнес-правилам;
- получение ответа модели и сохранение в документ (в т.ч. DOCX-вложение);
- полный **аудит** обращений и понятные причины отказов;
- подготовка к **автоматизации** (схемы обработки, триггеры) и к **облачным** провайдерам без переписывания ядра.

---

## 2. Принципы и ограничения

| Принцип | Содержание |
|--------|------------|
| Граница изменений | Доработки — в расширении; типовую конфигурацию ДО не менять без решения заказчика. |
| Рабочий сценарий | Ручной запуск анализа из карточки **обязан** оставаться стабильным при любых доработках бэклога. |
| Зависимости среды | Расширение `lvv_ОтклБезопасныйРежимВыполненияСкриптов` и сценарии **скриптов схем** — отдельная линия; в плане явно отмечено как отложенное / вне ядра ИИ. |
| Безопасность данных | Учёт политики выноса текста за контур; для облака — ключи и секреты только через БСП/утверждённые механизмы. |

## 4. Стратегический план (фазы)

Фазы упорядочены по **зависимостям**: сначала доказуемая стабильность и наблюдаемость, затем UX и производительность, затем рефакторинг и масштабирование, затем облако и глубокая автоматизация.

### Фаза A — Стабилизация и доказательная база (P0)

**Цель:** зафиксировать «что считаем рабочим» и границы по объёму/типам файлов.

| ID | Задача | Критерий готовности |
|----|--------|---------------------|
| A1 | Интеграционные прогоны на типах файлов: PDF текстовый, PDF скан, DOCX, длинный текст (>200k символов) | Протокол/чек-лист в `docs/` или wiki; зафиксированы дефекты и ограничения |
| A2 | Проверка открытия сформированного DOCX в Word/LibreOffice | Минимум 2 клиента; список известных ограничений формата |
| A3 | SLA таймаутов: HTTP к Ollama, ожидание распознавания, общий сценарий | Таблица значений по умолчанию + что выносится в настройки (см. B1) |

**Зависимости:** нет.  
**Риски:** сканы и длинные документы выявляют лимиты контекста модели — зафиксировать как продуктовое ограничение или завести задачу на чанкинг (фаза C).

---

### Фаза B — Эксплуатация и прозрачность (P1)

**Цель:** администратор и пользователь понимают состояние и причину отказа без разбора кода.

| ID | Задача | Критерий готовности |
|----|--------|---------------------|
| B1 | Таймаут ожидания распознавания — константа или регистр/параметр | Значение настраивается без изменения кода; описано в инструкции |
| B2 | UI карточки документа: статус «анализ в процессе / ожидается распознавание / ошибка» | Пользователь видит состояние без открытия журнала |
| B3 | Журнал: **коды/строки причин** отказа (нет прав, пустой текст, провайдер недоступен, уже в очереди, таймаут и т.д.) | Фильтрация и разбор инцидентов по полю причины |
| B4 | Корреляционный идентификатор цепочки (распознавание → запрос к модели → файл ответа) в журнале | По одному GUID видна вся цепочка в регистре |

**Зависимости:** A3 желательна до B1/B2 (согласованные дефолты).  
**Риски:** избыточный шум в журнале — договориться о перечне кодов и уровнях детализации.

---

### Фаза C — Производительность и устойчивость интеграции

**Цель:** предсказуемое поведение под нагрузкой и при сбоях сети/Ollama.

| ID | Задача | Критерий готовности |
|----|--------|---------------------|
| C1 | Единый слой HTTP: ретраи при транзиентных ошибках (лимит попыток, пауза), единые лимиты логирования тела ответа | Поведение описано в комментарии к модулю; нет дублирования в каждом вызове |
| C2 | Ограничение параллелизма запросов к Ollama (параметр/семафор) | Нагрузочный сценарий не «убивает» сервер моделей |
| C3 | Политика **длинных документов**: чанкинг, сжатие контекста или явный отказ с понятным сообщением | Задокументировано в UX и в журнале |
| C4 | (Опционально) Circuit breaker: после серии ошибок — пауза и код состояния «провайдер недоступен» | Снижение шума фоновых заданий |

**Зависимости:** A1 помогает приоритизировать C3.  
**Риски:** чанкинг меняет смысл «одного ответа» — согласовать с бизнесом (один файл vs несколько блоков).

---

### Фаза D — Архитектура кода и техдолг

**Цель:** снизить стоимость сопровождения и повторного использования логики.

| ID | Задача | Критерий готовности |
|----|--------|---------------------|
| D1 | Декомпозиция `ИАДО_ИнтеграцияИИВызовСервера`: подбор файла/документа, очередь/распознавание, вызов модели/постобработка | Модули с границами ответственности; без регрессии ручного сценария |
| D2 | Устранение предупреждений документирования BSL (`constructor-function-return-section` и др.) в затронутых модулях | `syntaxcheck`/EDT без критичных замечаний по стилю |
| D3 | Ревизия **опциональных** заимствований (`ОбработкиОбъектов`, `ManagerModule` документов): либо поддержка UUID-реестра для команд, либо удаление мёртвых входов | В `INSTRUCTIONS` одна каноническая строка для сценариев с `lvv` |
| D4 | Smoke-проверка целостности BSL (нет мусора в конце файлов); при необходимости шаг CI/скрипт | Регламент в `docs/` |

**Зависимости:** можно параллелить с B после A1.  
**Риски:** большой рефакторинг — только с тестовым прогоном из фазы A.

### Фаза E — Автоматизация жизненного цикла документа (отложено отдельной инициативой)

**Статус:** **приостановлено** (скрипт схемы обработки объекта — до выяснения политики расширений и `lvv`).

| ID | Задача | Примечание |
|----|--------|------------|

| E2 | Триггеры по этапам ДО (регистрация, состояние, появление версии файла) | Идемпотентность, фон, без блокировки основного процесса |
| E3 | Уведомления пользователю об ошибках/завершении автоматического анализа | После E1–E2 |

**Зависимости:** решение заказчика по связке расширений; при необходимости отдельный PRD.

### Фаза F — Облачные ИИ

**Цель:** второй провайдер рядом с Ollama без дублирования бизнес-логики.

| ID | Задача | Критерий готовности |
|----|--------|---------------------|
| F1 | Абстракция провайдера (интерфейсный модуль или стратегия по перечислению провайдера) | Ollama остаётся эталонной реализацией |
| F2 | Хранение ключей (БСП), таймауты, лимиты, маскирование в журнале | Соответствие ИБ-политике |
| F3 | Выбор провайдера в эксперте/правиле | Согласовано с журналом и UI |

**Зависимости:** фазы A–B желательны (прозрачность ошибок и настройки).  
**Риски:** правовой и ИБ-аудит выноса данных.
